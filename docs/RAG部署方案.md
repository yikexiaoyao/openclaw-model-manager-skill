# 📚 RAG 自动部署方案

**创建时间：** 2026-04-07  
**目标：** 为 OpenClaw 部署本地 RAG（检索增强生成）系统

---

## 🎯 一、RAG 是什么？

### 核心概念

```
RAG = Retrieval（检索）+ Augmented（增强）+ Generation（生成）
```

**通俗解释：**

> 没有 RAG：AI 只靠训练时的知识回答（可能过时/瞎编）
> 
> 有 RAG：AI 先从你的文档里找相关信息，再基于这些信息回答（准确/可追溯）

---

## 🔧 二、RAG 工作原理

### 完整流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        RAG 工作流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【索引阶段】                                                    │
│  你的文档 → 分块 → 嵌入模型 → 向量 → 存入向量数据库              │
│                                                                 │
│  【查询阶段】                                                    │
│  用户问题 → 嵌入模型 → 问题向量 → 向量数据库检索 → 相关文档块    │
│              ↓                                                  │
│          Reranker 重排序 → 最相关的 Top K                        │
│              ↓                                                  │
│          LLM + 问题 + 相关文档 → 准确回答                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 举个例子

**用户问：** "我们项目的 API 密钥存在哪里？"

**没有 RAG：**
> AI 瞎编一个答案，或者承认不知道

**有 RAG：**
1. 把你的问题转成向量
2. 从向量数据库里找到最相关的文档片段：
   - `docs/config.md`: "API 密钥存储在 `~/.openclaw/.env` 文件中"
   - `docs/security.md`: "不要将密钥提交到 Git"
3. 把这些片段 + 问题一起给 LLM
4. LLM 回答："API 密钥存储在 `~/.openclaw/.env` 文件中，注意不要提交到 Git"

## 🧩 三、核心组件详解

### 1️⃣ 嵌入模型 (Embedding Model)

**作用：** 把文字转成向量（数字列表）

| 模型 | 维度 | 中文支持 | 速度 | 内存 |
|------|------|---------|------|------|
| **BGE-M3** | 1024 | ⭐⭐⭐⭐⭐ | 快 | ~1 GB |
| **BGE-Large-ZH** | 1024 | ⭐⭐⭐⭐⭐ | 中 | ~2 GB |
| **m3e-base** | 768 | ⭐⭐⭐⭐⭐ | 快 | ~500 MB |
| **text-embedding-3-small** (OpenAI) | 1536 | ⭐⭐⭐⭐ | API | $ |

**推荐：BGE-M3**
- 多语言支持好
- 支持长文本（8192 tokens）
- 本地运行，免费

---

### 2️⃣ 向量数据库 (Vector Database)

**作用：** 存储和检索向量

| 数据库 | 特点 | 内存占用 | 适合场景 |
|--------|------|---------|---------|
| **ChromaDB** | 轻量、易部署 | ~500 MB | ✅ 个人/小项目 |
| **FAISS** (Facebook) | 超快、纯内存 | ~1-2 GB | 大规模检索 |
| **Qdrant** | 功能全、可集群 | ~1 GB | 生产环境 |
| **Pinecone** | 云服务 | - | 不想自己部署 |

**推荐：ChromaDB**
- Python 原生，部署简单
- 持久化存储（重启不丢数据）
- 支持元数据过滤

---

### 3️⃣ Reranker (重排序模型)

**作用：** 对检索结果二次排序，提高精度

**为什么需要？**
- 向量检索是"粗筛"，可能不够准
- Reranker 仔细比较问题和文档的相关性
- 可以把最相关的排到前面

| 模型 | 中文支持 | 速度 | 内存 | 精度提升 |
|------|---------|------|------|---------|
| **BGE-Reranker-Large** | ⭐⭐⭐⭐⭐ | 快 | ~1 GB | +15-20% |
| **BGE-Reranker-Base** | ⭐⭐⭐⭐⭐ | 快 | ~500 MB | +10-15% |
| **Cohere Rerank** (API) | ⭐⭐⭐⭐ | API | - | +15% |

**推荐：BGE-Reranker-Base**
- 中文效果好
- 内存占用低
- 本地运行，免费

---

### 4️⃣ LLM (大语言模型)

**作用：** 基于检索结果生成回答

**你已经有了：**
- ✅ **Qwen3.5-9B-MLX** (本地，8000 端口)

**可选升级：**
- Qwen3.5-27B (云端，更强大)
- 其他 API 模型

## 🛠️ 六、部署步骤（方案 B）

### 步骤 1：安装 Python 依赖

```bash
pip install chromadb sentence-transformers flagembedding
```

**预计时间：** 5-10 分钟  
**下载大小：** ~2 GB

---

### 步骤 2：下载嵌入模型

```bash
# BGE-M3 会自动下载
# 位置：~/.cache/huggingface/hub/
```

**模型大小：** ~1 GB  
**下载时间：** 取决于网速

---

### 步骤 3：下载 Reranker 模型

```bash
# BGE-Reranker-Base 会自动下载
```

**模型大小：** ~500 MB

---

### 步骤 4：启动 ChromaDB 服务

```bash
chroma run --path ~/.openclaw/chroma --port 8001
```

---

### 步骤 5：安装 RAG 技能

```bash
clawhub search "rag"
clawhub install <选中的 RAG 技能>
```

---

### 步骤 6：配置 OpenClaw

在 `~/.openclaw/openclaw.json` 中添加：

```json
{
  "rag": {
    "enabled": true,
    "embedding_model": "BAAI/bge-m3",
    "chroma_host": "http://localhost:8001",
    "reranker_model": "BAAI/bge-reranker-base",
    "top_k": 5
  }
}
```

## 📊 七、使用场景

### 适合用 RAG 的场景

| 场景 | 说明 |
|------|------|
| 📁 **个人文档问答** | "我上周写的会议纪要在哪？" |
| 💻 **代码库问答** | "这个项目的认证逻辑在哪？" |
| 📚 **知识库问答** | "公司报销流程是什么？" |
| 🔍 **长文档检索** | "合同里关于违约的条款" |
| 📝 **笔记关联** | "我之前记过类似的想法吗？" |

### 不适合 RAG 的场景

| 场景 | 原因 |
|------|------|
| 闲聊 | 不需要检索 |
| 通用知识 | LLM 自己知道 |
| 实时信息 | 需要搜索，不是 RAG |
| 数学计算 | 检索帮不上忙 |

## 🎯 八、效果预期

### 索引速度

|---------|------|

## ⚠️ 九、注意事项

### 1. 隐私安全

- ✅ 所有数据本地存储
- ✅ 不上传云端
- ⚠️ 注意备份向量数据库

### 2. 内存管理

- 首次加载模型占用高
- 可以设置自动卸载（空闲时）
- 建议保留 2 GB 系统缓冲

### 3. 数据更新

- 新增文档需要重新索引
- 可以设置自动监听文件变化
- 删除文档需要手动清理向量

## 🎬 十、决策建议

### 建议部署，如果：

- ✅ 你有大量本地文档需要检索
- ✅ 你在维护大型代码库
- ✅ 你想让 AI 更了解你的项目
- ✅ 你有 10 GB+ 可用内存

### 可以暂缓，如果：

- ⏸️ 你主要用 AI 闲聊
- ⏸️ 你的文档都在云端（Notion 等）
- ⏸️ 内存紧张（<8 GB 可用）
